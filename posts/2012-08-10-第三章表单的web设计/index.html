我们在第一章和第二章看到了，如合使用控件和验证规则创建表单。

<span en=“We used the 〈?php echo $form ?〉 statement to display them.”>我们用了 <?php echo $form ?> 语句来显示它们。

当开发者把注意力集中在模型和业务逻辑上时，每次更改模板只修改一个字段（名字、控件…），甚至添加一个不必要的，这个语句允许开发者编写应用逻辑而不用考虑最终外观是如何的，这个语句适合于原型和初始开发阶段。

一旦对象模型稳定了，风格样式固定了，WEB设计师就可以开始设计制作各个应用的表单。

3.3 原型模板的定制

<span en=“The 〈?php echo $form ?〉 statement can be enough for simple forms like the contact form.”><?php echo $form ?> 语句对于简单表单比如留言板表单足够了。

<span en=“And, as a matter of fact, it is just a shortcut for the 〈?php echo $form->render() ?〉 statement.”>事实上，它只是 <?php echo $form->render() ?> 方法的快捷语句。

<span en=“The usage of the render() method allows to pass on HTML attributes as an argument for each field.”>对于每个字段的 render() 方法允许传递HTML属性作为一个参数。

<span en=“Listing 3-3 shows how to add a class to the email field.”>例3-3展示如何添加一个 class 到 email 字段。

3.4.7 处理总体错误

对于一个表单，有3种错误提示：

  • 关联到指定字段的错误提示

  • 总体错误提示

  • 错误来自隐藏字段或字段，不实际在表单中显示。

    那是使用全局错误归纳起来的

We already went over the implementation of error messages associated with a field, and Listing 3-15 shows the implementation of global error messages.

<span en=“The call to the renderGlobalErrors() method displays the global error list.”>要调用 renderGlobalErrors() 方法显示总体错误提示列表。

<span en=“It is also possible to access the global errors using the hasGlobalErrors() and getGlobalErrors() methods, as shown in Listing 3-16.”>它还可以用 hasGlobalErrors() 和 getGlobalErrors() 方法访问，如例3-16所示。

<span en=“Each global error has a name (name) and a message (error).”>每个总体错误都有一个名字（name）和一个消息（error）。

The name is empty when there is a “real” global error, but when there is an error for a hidden field or a field that is not displayed, the name is the field label name.

3.6 协作开发

让我们用典型的表单开发情景的描述来结束本章：

  • 开发团队开始实现表单类以及行为。

    <span en=“The template is basically nothing more than the 〈?php echo $form ?〉 prototyping statement.”>模板基本上无非就是写一句 <?php echo $form ?>。

  • 在此期间，设计师设计风格样式，并应用到表单上：总体结构、错误提示显示规则等等。

  • 一旦业务逻辑还有风格样式固定了，WEB设计团队就可以修改表单模板并定制。

    WEB设计团队只需要知道字段的名字，还有行为（action）要处理的表单的生命周期。

当第一个周期结束，业务规则修改和模板修改也就都完成了。

而不影响模板，所以不需要设计团队的介入，开发团队就可以了：

  • 修改表单的控件
  • 定制错误提示
  • 编辑、添加或删除验证规则

Likewise, the designer team is free to perform any ergonomic or graphic changes without falling back on the development team.

但下面这些涉及到两个团队间协调：

  • 重命名一个字段
  • 添加或删除一个字段

This cooperation makes sense as it involves changes in both business rules and form display.

Like we stated at the beginning of this chapter, even if the form system cleanly separates the tasks, there is nothing like communication between the teams.
