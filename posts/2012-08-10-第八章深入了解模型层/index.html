到目前为止本书大多数的讨论都专注于建立页面，处理请求与响应。

但是web应用的业务逻辑大多依赖于它的数据模型。

symfony的默认模型组件是基于一个对象/关系映射层。

<span en=“Symfony comes bundles with the two most popular PHP ORMs: <a target="_blank” href="http://www.propelorm.org/">Propel and &lta target="_blank" href="http://www.doctrine-project.org/">Doctrine.“>symfony捆绑两个最流行的PHP ORM：Proopel 和 Doctrine。

在symfony应用中，你不用关注数据库的实际位置，而是通过对象来访问储存在数据库中的数据的。

这保持了高度的抽象和可移植性。

本章解释了如何建立一个数据对象模型和如何通过Doctrine来访问和修改数据。

同时也展示了symfony是如何整合Doctrine的。

8.1 为什么使用ORM和抽象层？

数据库是关系型的。

PHP5和symfony都是面向对象的。

为了在面向对象环境中最有效的访问数据库，一个接口转换“对象逻辑”为“关系逻辑”是必须的。

如第一章所述，这个接口就叫做对象-关系映射（ORM），它是由可以访问数据和保持业务规则的对象组成的。

ORM最大的优势就是可重用性，它允许数据对象的方法可以被应用的其他部分调用，甚至可以从不同的应用中调用。

ORM层也可以封装数据逻辑，例如，基于用户的贡献度和如何作出的贡献来计算论坛用户的评分。

当一个页面需要显示例如一个用户的评分，不需要担心如何去计算，只要很简单的调用数据模型的方法即可。

如果以后计算方法有所变化，你只需要修改模型的评分方法即可，应用的其他部分不需要改变。

使用对象来代替记录，用类来代替表，还有其他好处：他们允许你在对象中增加一个新的读取方法而不需要对应到表的一个列。

<span en=“For instance, if you have a table called client with two fields named first_name and last_name, you might like to be able to require just a Name.”>例如，如果你有一个 client 表，它拥有两个字段分别叫做 first_name 和 last_name，你可能只想要获得一个 Name。

<span en=“In an object-oriented world, it is as easy as adding a new accessor method to the Client class, as in Listing 8-1.”>在面向对象的世界里，Client 类中增加一个存取方法是非常简单的，如例8-1所示。

<span en=“From the application point of view, there is no difference between the FirstName, LastName, and Name attributes of the Client class.”>从应用的角度来看，Client 类的 FirstName、LastName 和 Name 属性没有什么区别。

只有类本身才能决定属性所对应的数据库的列。

所有重复的数据访问函数和数据自身的业务逻辑可以存在对象中。

<span en=“Suppose you have a ShoppingCart class in which you keep Items (which are objects).”>假设你有一个 ShoppingCart 类，里面有一个 Item（是个对象）。

只要写一个自定义的方法来封装实际的计算过程，就可以在结账时得到购物车的总价。

如例8-2所示。

在建立数据访问过程的时候还要考虑另外一个要点：数据库厂商所使用的不同的SQL语法变种。

换用另外一个数据库管理系统（DBMS）会让你不得不重写一部分为以前设计的SQL查询。

如果用数据库独立语法来建立一个查询，并把实际SQL翻译为第三方语言，换数据库系统就不会麻烦了。

这就是数据抽象层存在的目的。

它强制让你使用特定的语法规则来写查询，同时把它转到到相应的DBMS并优化SQL语句。

抽象层的主要优势是可移植性，因为他让换用另一种数据库成为可能，甚至可以在项目中期换用。

假设你需要为应用写一个快速原型，但客户还没有确定哪个数据库最适合他。

你可以先用SQLite建立你的应用，然后在客户有了决定后切换到MySQL、PostgreSQL或者Oracle。

这只要改变配置文件中一行代码即可。

symfony使用Propel或者Doctrine来实现ORM，然后用PDO（PHP Data Object）做数据库抽象。

这两个第三方组件，都是由Propel和Doctrine团队开发的，并且都无缝集合到了symfony中，你可以把他们作为框架的一部分。

在本章描述的这些语法和规则与symfony框架本身的语法的可能会有点不同。

8.2 Symfony的数据库schema

为了创建symfony使用的数据对象模型，需要把数据库关系模型翻译为对象数据模型。

ORM需要关系模型的描述来做映射，这就叫做schema。

在schema中，你定义表和表之间的关系，还有表中字段列的详细。

symfony中的schema语法使用YAML格式。

<span en=“The schema.yml files must be located in the myproject/config/doctrine directory.”>schema.yml 文件必须放在 myproject/config/doctrine 目录下。

8.3 Schema示例

如何把数据库结构转换为schema呢？看例子是最好的理解方法。

<span en=“Imagine that you have a blog database with two tables: blog_article and blog_comment, with the structure shown in Figure 8-1.”>试想一下你有一个博客数据库，包含两个表 blog_article 和 blog_comment，结构如图8-1所示。

<span en=“The related schema.yml file should look like Listing 8-3.”>对应的 schema.yml 文件应该看上去如例8-3所示。

<span en=“Notice that the name of the database itself (blog) doesn’t appear in the schema.yml file.”>注意数据库本身的名字（blog）并没有出现在 schemal.yml 文件中。

<span en=“Instead, the database is described under a connection name (doctrine in this example).”>反而在连接名下有数据库描述（本例中是 doctrine）。

这是因为实际的连接设置可以依照应用运行的环境来定。

<span en=“For instance, when you run your application in the development environment, you will access a development database (maybe blog_dev), but with the same schema as the production database.”>例如，当你在开发环境中运行应用时，你会访问一个开发数据库（比如是 blog_dev），但是产品数据库也用的是同一个schema文件。

<span en=“The connection settings will be specified in the databases.yml file, described in the “Database Connections” section later in this chapter.”>在 database.yml 文件中有连接，本章稍后的“数据库连接”会有介绍。

schema不包含任何连接设置的细节，只有连接名用来保持数据库抽象。

8.3.1 Schema基本语法

<span en=“In a schema.yml file, the first key represents a model name.”>在 schema.yml 文件中，第一个关键字代表一个模型名。

你可以指定多个模块，每个模型都有一组字段列。

根据YAML语法，关键字用冒号作结束标记，然后结构是通过缩进显示的（一个或多个空格，但不是制表符）。

<span en=“A model can have special attributes, including the tableName (the name of the models database table).”>模型可以有一些属性，包括 tableName（模型数据表的名称）。

<span en=“If you don’t mention a tableName for a model, Doctrine creates it based on the underscored version of the model name.”>对于模型如果你没有设置 tableName，Doctrine则根据模型名字的下划线命名法自动创建它。

一个模型包含许多字段列。

有两种方法可以定义字段：

  • 如果只定义一个属性，则为字段类型。

    <span en=“Symfony understands the usual column types: boolean, integer, float, date, string(size), clob (converted, for instance, to text in MySQL), and so on.”>symfony支持一些常用的字段类型：boolean、integer、float、date、string(size)、clob（转换过的，例如MySQL中的 text），等等。

  • <span en=“If you need to define other column attributes (like default value, required, and so on), you should write the column attributes as a set of key: value.”>如果需要定义字段列的其他属性（如默认值、是否为必填等等），你需要把字段属性写为一组 key: value。

    这种扩展的schema语法稍后在本章介绍。

模型还可以包含详细外键和索引。

请参阅“扩展Schema语法”这节，在本章后面会了解到更多。

8.4 模型类

Schema用于建立ORM层的模型类的。

<span en=“To save execution time, these classes are generated with a command-line task called <code.doctrine:build-model.”>为了省时，这些类是通过命令行调用 doctrine:build-model 来生成的。

$ php symfony doctrine:build-model

<span en=“Typing this command will launch the analysis of the schema and the generation of base data model classes in the lib/model/doctrine/base directory of your project:”>输入这个命令会加载schema并分析，然后在你项目的 lib/model/doctrine/base 目录生成基础数据模型类。

  • BaseArticle.php
  • BaseComment.php

<span en=“In addition, the actual data model classes will be created in lib/model/doctrine:”>此外，会在 lib/model/doctrine 创建实际数据模型类：

  • Article.php
  • ArticleTable.php
  • Comment.php
  • CommentTable.php

你只定义了两个模型，而到6个文件。

没有错这是正常的，然后我解释一下。

8.4.1 基础和自定义类

为什么数据对象模型在两个不同的目录有两个不同的版本？

<span en=“You will probably need to add custom methods and properties to the model objects (think about the getName() method in Listing 8-1). “>你可能会需要添加自定义的方法和属性到模型对象（比如例8-1的 getName() 方法）。

但随着你项目的开发，你还会添加数据表或字段。

<span en=“Whenever you change the schema.yml file, you need to regenerate the object model classes by making a new call to doctrine:build-model.”>只要你修改了 schema.yml 文件，就要运行 doctrine:build-model 命令重新生成对象模型类。

如果你在自动生成出来的类上写你自己的自定义方法，那么以后每次再生成它们会被覆盖。

<span en=“The Base classes kept in the lib/model/doctrine/base/ directory are the ones directly generated from the schema.”>位于 lib/model/doctrine/base/ 目录里的 Base 类是来自schema直接生成的。

你不能修改它们，因为每次新建的模型会完全覆盖这些文件。

<span en=“On the other hand, the custom object classes, kept in the lib/model/doctrine directory, actually inherit from the Base ones.”>另一方面，自定义类，位于 lib/model/doctrine 目录里，实际上是继承自 Base 类。

<span en=“When the doctrine:build-model task is called on an existing model, these classes are not modified.”>如果 doctrine:build-model 命令调用了一个已有的模型的时候，这些类则不会被修改。

所以这个就是你可以添加自定义方法的地方。

<span en=“Listing 8-4 presents an example of a custom model class as created by the first call to the doctrine:build-model task.”>例 8-4 展示的是第一次调用 doctrine:build-model 命令建立的一个自定义模型类的例子

<span en=“It inherits everything of the BaseArticle class, but a modification in the schema will not affect it.”>它继承 BaseArticle 类的一切，但修改schema并不会影响到这个文件。

用自定义类来扩展基础类的机制可以让你在不知道最终数据库中模型之间的关系的时候开始编程。

相关的文件结构会让模型既可以自定义又可以进化。

8.4.2 对象和数据表类

<span en="Article and Comment are object classes that represent a record in the database.”>Article 和 Comment 是对象类，在数据库中代表一条记录。

他们可以访问到记录的字段类和有关记录。

这就是说你可以调用 Article 对象的方法来获取文章的标题，如例8-5所示。

<span en="ArticleTable and CommentTable are table classes; that is, classes that contain public methods to operate on the tables.”>ArticleTable 和 CommentTable 是数据表类；因此，类包含了公有的方法来操作数据表。

他们提供了从表中获得记录的方法。

他们的方法通常返回了一个对象或是相关对象类的对象的集合，如例8-6所示。
